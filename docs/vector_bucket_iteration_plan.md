# Vector Bucket 方案与迭代建议

## 1. 背景

当前要做的是一个类似云厂商 `vector bucket` 的产品能力，核心目标不是直接暴露 Milvus，而是提供一个更稳定、更简单的产品接口。

对用户暴露的接口应尽量简单：

- `insert / upsert / delete`
- `query(topK, filter)`

用户不应该感知底层到底是：

- `HNSW`
- `DISKANN`
- `IVF_PQ`

底层方案应该可以演进，但外部产品接口应尽量保持稳定。

本次讨论的重点不是一步到位做最终架构，而是：

- 先快速做出一个可用版
- 在不推翻用户接口的前提下持续迭代

## 2. 当前资源与约束

当前已知约束如下：

- 基础设施是 HCI 平台。
- 当前机器规格大约是 `8 vCPU / 16 GB RAM`。
- 这台机器同时承担对象存储服务。
- 对象存储可以通过 JuiceFS 提供 bucket。
- 当前也有一个非 JuiceFS 的目录可用。
- 查询目标 `topK` 较小，当前可按 `1-30` 设计。
- 长期希望同时支持低成本大规模索引，以及少量高性能热点索引。
- 当前更重要的是尽快做出可用版，而不是一开始就完成终态。
- 第一阶段不适合做大量 Milvus 内核深改。

这些约束比“理论上最优的 ANN 算法”更重要。

## 3. 产品目标

近阶段产品目标：

- 尽快做出可用服务
- 对外接口稳定
- 为后续更换底层索引方案留好空间
- 第一阶段避免进入高复杂度架构

中阶段产品目标：

- 保持逻辑索引抽象不变
- 支持后端方案升级
- 最终支持热点和冷数据分层，但不暴露给用户

## 4. 候选方案

### 4.1 方案 A：只做 `HNSW + load/release + LRU`

方案思路：

- 每个逻辑索引先映射到一个 Milvus collection 或 partition 集合
- 当前唯一后端索引类型使用 `HNSW`
- 查询前检查目标索引是否已经 `load`
- 未加载时按需 `load`
- 通过简单的 `LRU + TTL` 在空闲时 `release`

优点：

- 实现范围最小
- 最贴近当前想要的“首查慢，后续快”效果
- 语义简单，易于理解和调试
- 对小 `topK` 的查询质量更友好
- 第一版无需解决多后端路由问题

缺点：

- 内存开销高
- 规模上限直接受 RAM 约束
- 冷查询延迟包含 `load` 成本
- 还没有低成本全量承载层

评估：

- 最适合作为 V1

### 4.2 方案 B：直接使用 `DISKANN`

方案思路：

- 从第一版起，默认主后端就是 `DISKANN`
- 通过本地磁盘降低对 DRAM 的依赖

优点：

- 更接近“向量桶”长期产品形态
- 比纯内存 `HNSW` 更适合大于内存规模的数据
- 一般比 `IVF_PQ` 更适合作为产品默认质量方案

缺点：

- 很依赖本地 SSD 的随机读能力
- 当前机器与对象存储服务混部，I/O 风险较高
- 运维和调优复杂度高于 `HNSW`
- 不能自然提供当前最重视的“显式 load/release 缓存语义”
- 仍然不能单独解决热点和冷数据分层

评估：

- 适合作为后续迭代方向
- 不适合作为当前资源和节奏下的第一版默认方案

### 4.3 方案 C：直接使用 `IVF_PQ`

方案思路：

- 优先追求内存节省
- 用压缩索引作为默认承载层

优点：

- DRAM 占用最低
- 更容易在当前机器上容纳更大数据量
- 适合作为冷层或容量层

缺点：

- recall 上限较低
- 参数敏感
- 在 `topK <= 30` 且产品质量敏感的场景下，不适合作为唯一默认方案
- 不同数据集之间的效果波动更大

评估：

- 可以作为容量导向的后端
- 不建议作为第一版默认产品体验

### 4.4 方案 D：`HNSW` 热层 + `DISKANN` 基础层

方案思路：

- 热点数据放 `HNSW`
- 全量数据由 `DISKANN` 承载
- 查询优先打热层，不足时回基础层

优点：

- 最像长期正确的产品形态
- 兼顾质量、规模和内存可控
- 与期望的冷热分层能力最匹配

缺点：

- 实现复杂度明显更高
- 需要解决多后端元数据、路由、后台构建、迁移、切换
- 不适合作为第一版

评估：

- 适合作为长期目标架构
- 不适合作为 V1

## 5. 推荐结论

### 5.1 V1 推荐方案

推荐第一版采用：

- `HNSW`
- `load/release`
- `LRU + TTL`
- 一个逻辑索引抽象层

这个方案和最初的直觉是一致的，而且在综合比较之后，它仍然是最适合当前阶段的方案。

原因不是 `HNSW` 是最终答案，而是：

- 它最快能做出可用版
- 它最贴合当前想要的缓存语义
- 它能保留后续演进到 `DISKANN` 的空间

### 5.2 为什么当前不推荐直接做大方案

当前不推荐一开始就做 `DISKANN + HNSW` 双层架构，原因有四个：

- 第一版最重要的是把产品接口和使用链路跑通
- 当前机器资源不宽裕，而且对象存储服务混部
- `DISKANN` 的收益依赖真实本地 SSD 随机读能力
- 双层架构的复杂度远高于单后端版本

换句话说：

- V1 应该优先优化交付速度和可用性
- V2/V3 再优化规模和成本

## 6. 推荐的 V1 架构

第一版推荐架构：

```text
Client
  -> Gateway / API
  -> Metadata Service
  -> Milvus Adapter
  -> HNSW Backend
  -> LRU / TTL Cache Manager
```

核心行为：

- 用户只看到逻辑索引
- 每个逻辑索引映射到一个当前激活的 Milvus collection
- 首次查询可能触发 `load`
- 后续查询直接命中已加载的 `HNSW`
- 空闲索引由 `TTL` 或 `LRU` 回收

这套架构是刻意保持简单的。

第一版明确不做：

- 多后端联合查询
- 热层和基础层 merge
- 后台跨后端迁移
- segment 级多 profile 生命周期

## 7. 资源评估

### 7.1 当前机器预算

当前机器预算大致是：

- `8 vCPU`
- `16 GB RAM`
- 对象存储服务与 Milvus 共机部署

因此不能把 `16 GB` 全部视为向量检索可用内存。
实际规划时必须假设：

- 只有部分内存可稳定给搜索层使用
- page cache 与对象存储服务会争抢资源

### 7.2 `HNSW` 预留 4 GB 的粗略规模

如果 V1 给 `HNSW` 热层预留约 `4 GB` 内存，可以按下面的量级做安全规划：

| 向量维度 | 4 GB HNSW 预算下的粗略热点规模 |
| --- | --- |
| `768D float32` | 约 `60 万 - 90 万` |
| `1024D float32` | 约 `45 万 - 70 万` |
| `1536D float32` | 约 `30 万 - 50 万` |

这些数字不是硬保证，只是产品规划量级。
实际还会受到以下因素影响：

- `M`
- `efConstruction`
- metadata 大小
- partition 切分方式
- filter 字段数量
- 并发压力

### 7.3 未来 `DISKANN` 作为基础层时的粗略规模

如果后续在同一节点上增加 `DISKANN` 作为基础层，可按下面的量级做保守规划：

| 向量维度 | 混部场景下的粗略可检索活跃规模 |
| --- | --- |
| `768D float32` | 约 `200 万 - 400 万` |
| `1024D float32` | 约 `150 万 - 300 万` |
| `1536D float32` | 约 `100 万 - 200 万` |

这些估算成立的前提是：

- 非 JuiceFS 目录确实是本地磁盘路径
- 随机读延迟可接受
- 参数调优合理
- 不要求混部下极其稳定的尾延迟

如果这个“本地目录”背后不是足够快的本地 SSD，那么这些数字会偏乐观。

## 8. 工作量评估

### 8.1 V1：`HNSW + LRU load/release`

工作范围：

- 逻辑索引元数据
- Milvus 适配层
- create/load/release/search/delete 流程
- `LRU + TTL` 缓存管理
- 基础指标和失败处理

工作量预估：

- PoC：`1-2 周`
- 可用的内部版本：`2-4 周`
- 带基础观测和清理能力的版本：`4-6 周`

这个估算建立在以下假设上：

- 1 名工程师
- 直接使用 Milvus 现有能力
- 不修改 Milvus 内核主链路

### 8.2 V2：后端抽象与可替换激活后端

工作范围：

- 正式定义 backend 接口
- 一个逻辑索引保持只有一个激活后端
- 支持离线重建和切换到另一种后端

工作量预估：

- 在 V1 之后追加 `2-4 周`

这个阶段仍然不做多后端联合查询。

### 8.3 V3：`HNSW` 热层 + `DISKANN` 基础层

工作范围：

- 热层和基础层元数据
- 查询路由与 merge
- 回退搜索路径
- 后台构建与切换
- 热点提升和冷却回收

工作量预估：

- 在 V2 之后追加 `4-8 周`

### 8.4 如果选择深改 Milvus 原生支持多 profile

如果目标变成：

- 一个逻辑 field
- 多个物理索引 profile
- Milvus 内核原生切换

那么工作量会明显上升。

工作量预估：

- 约 `3-6 个月`
- 更适合多人协作，不适合作为第一阶段

## 9. 迭代方案

### Phase 1：先交付一个真正可用的版本

交付内容：

- 逻辑索引 API
- 唯一后端 `HNSW`
- `load/release`
- `LRU + TTL`
- 固定内存预算
- 基础监控指标

明确不交付：

- `DISKANN`
- `IVF_PQ`
- 热层和基础层双路查询
- 自动后端切换
- Milvus 深改

阶段目标：

- 跑通 insert/query/delete
- 内存行为可控
- 冷启动可接受
- 热点查询显著变快

### Phase 2：为后续替换后端做准备

交付内容：

- backend 抽象接口
- 离线重建与切换流程
- 保持外部 API 不变

阶段目标：

- 后续引入新 backend 时，不需要重做产品接口

### Phase 3：引入更大规模后端

交付内容：

- 增加 `DISKANN` backend
- 一个逻辑索引同一时刻只使用一个激活 backend
- 支持离线迁移或受控切换

阶段目标：

- 在不暴露底层方案给用户的前提下支持更大数据量

### Phase 4：增加冷热分层

交付内容：

- `HNSW` 热层
- `DISKANN` 基础层
- 路由和 merge

阶段目标：

- 接近长期向量桶产品架构

## 10. 风险

V1 主要风险：

- 冷查询延迟较高，因为 `load` 在请求路径上
- `HNSW` 在真实数据上的内存占用可能高于预估
- 如果租户分布不均，collection 级缓存粒度可能偏粗

后续阶段主要风险：

- `DISKANN` 的收益强依赖真实本地磁盘质量
- 与对象存储服务混部可能导致尾延迟不稳定
- 热层和基础层架构的生命周期复杂度明显更高

## 11. 最终建议

建议当前做法是：

- 第一版采用 `HNSW + LRU load/release`
- 给 `HNSW` 预留约 `4 GB` 预算
- 从一开始就把产品接口固定成逻辑索引接口
- 在代码层面预留 backend 抽象，但第一版只实现一个 backend
- `DISKANN` 放到后续迭代，在验证了 API、缓存语义和运行模式后再接入

这条路线在以下几个维度上最均衡：

- 交付速度
- 产品可用性
- 后续演进空间
- 当前机器资源约束

最重要的一点是：

- `HNSW + LRU` 不是最终架构
- 但它是当前最实际的第一版架构
